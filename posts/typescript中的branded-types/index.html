Typescript中的Branded Types

2026/2/25 00:23:4421 分钟阅读
Typescript

问题背景

以一个简单的 用户-博文 系统为例：

interface User {}
interface Post {}

type UserId = string;
type PostId = string;

declare function getUser(id: UserId): User;
declare  function getPost(id: PostId): Post;

const userId: UserId = "user_123";
const postId: PostId = "post_123";

// 虽然TypeScript编译通过，但从业务上说，逻辑是错误的!
getUser(postId); // Compiles, Runtime bug: fetches wrong data
getPost(userId); // Compiles, Runtime bug: fetches wrong data

可以看到，UserId和PostId本质上都是string类型，对于这个基本类型，Typescript 无法知道这个基本类型在不同结构类型中是需要作区分的，导致我们在后续代码中可能错误地传递参数，且 Typescript 仍可编译通过无报错，进而引发运行时的错误。

引入Branded Types

Branded types，别名tagged types或opaque types，是一种在结构类型语言中创建名义上不同的类型的技术。

核心思想

其基本原理很简单：给类型添加一个独特的标记，使其在名义上与众不同，即使底层结构相同。该技术利用 Typescript 的交叉类型来添加一个幻影属性-- 一个只在编译时存在，运行时永远不会存在的属性。

基本使用

还是上面那个例子，我们稍作改造：

interface User {}
interface Post {}

type UserId = string & { readonly __brand: "UserId" };
type PostId = string & { readonly __brand: "PostId" };


declare function getUser(id: UserId): User;
declare  function getPost(id: PostId): Post;

const userId = "user_123" as UserId;
const postId = "post_123" as PostId;

// 编译提示错误
getUser(postId);
/**
 * Argument of type 'PostId' is not assignable to parameter of type 'UserId'.
  Type 'PostId' is not assignable to type '{ readonly __brand: "UserId"; }'.
    Types of property '__brand' are incompatible.
      Type '"PostId"' is not assignable to type '"UserId"'.(2345)
 */
getPost(userId);
/**
 * Argument of type 'UserId' is not assignable to parameter of type 'PostId'.
  Type 'UserId' is not assignable to type '{ readonly __brand: "PostId"; }'.
    Types of property '__brand' are incompatible.
      Type '"UserId"' is not assignable to type '"PostId"'.(2345)
 */

现在，Typescript 可以把UserId和PostId视作不同的类型，即使他们在底层结构上都是string类型。{ readonly __brand: "UserId" }和{ readonly __brand: "PostId" }使得这两个类型不再相互兼容。

Warning

注意：在运行时，UserId和PostId仍然只是普通的字符串，__brand属性在编译期间便被消除。这意味着：

  • 零运行时开销
  • 零额外性能开销
  • 对内存没有影响
  • 完全编译期安全

进阶使用

每次写string & { readonly __brand: "UserId" }有些过于硬核，我们可以创建一些辅助类型用于生成Branded types。

使用泛型

type Brand<T, BrandName extends string> = T & { readonly __brand: BrandName };

这个Brand类型接受两个参数：

  • T: 底层类型（如string、number等）
  • BrandName: 一个唯一字符串字面量用于定义和区分brand

现在，我们可以这样子创建Branded types:

type UserId = Brand<string, "UserId">;
type PostId = Brand<string, "PostId">;
type Email = Brand<string, "Email">;

type Meters = Brand<number, "Meters">;
type Feet = Brand<number, "Feet">;
type USD = Brand<number, "USD">;

简洁了许多，但我们可以再进一步优化。

唯一symbol

为了更高的安全性，我们可以使用 Typescript 的unique symbol特性取代字符串字面量。

declare const brandSymbol: unique symbol;

type Brand<T, BrandName> = T & { [brandSymbol]: BrandName };

// 用法相同
type UserId = Brand<string, "UserId">;
type PostId = Brand<string, "PostId">;
type Email = Brand<string, "Email">;

使用unique symbol有两个优势：

  • 防止意外的类型匹配。使用像 __brand 这样的基于字符串的品牌标识时，存在一个风险：来自第三方库的实际对象可能确实具有名为 __brand 的属性。使用唯一符号可以完全消除这种风险——符号保证是唯一的，并且该标识仅存在于类型级别。
  • 更友好的IDE体验。当你使用像 __brand 这样的字符串属性时，它会在你输入 someObj 时出现在 IDE 的自动补全功能中，这会对编程造成干扰。而symbol不会出现在自动补全功能中，从而保持开发体验的简洁。

简短的构造函数

由于Branded types的属性在运行时并不存在，我们自然无法直接使用下面的方式创建Branded types:

type UserId = string & { readonly __brand: "UserId" };

const userId_1: UserId = { __brand: "UserId" };
/**
 * Type '{ __brand: "UserId"; }' is not assignable to type 'UserId'.
  Type '{ __brand: "UserId"; }' is not assignable to type 'string'.(2322)
*/
const userId_2: UserId = "user_123";
/**
 * Type 'string' is not assignable to type 'UserId'.
  Type 'string' is not assignable to type '{ readonly __brand: "UserId"; }'.(2322)
*/

因此，我们可以使用一个简单的构造函数来创建Branded types并减少as的滥用：

type UserId = Brand<string, "UserId">;

// 不推荐: 类型断言随处散落
const userId1: UserId = "1234" as UserId;
const userId2: UserId = "5678" as UserId;
const userId3: UserId = "9012" as UserId;

// 推荐做法: 单一简短且具有语义化的构造函数
function createUserId(id: string): UserId {
  return id as UserId;
}

const userId1 = createUserId("1234");
const userId2 = createUserId("5678");
const userId3 = createUserId("9012");

值类型转换

上面提到的代码，均是传递已知为Branded types的值。但我们如何告诉 Typescript 某个新值是Branded types？ 这通常通过两种方式完成：类型断言或类型收缩。

as断言

利用as关键字确保 Typescript 知道某个新值是对应的类型：

type Positive = number & { __brand: "positive" };

let myPositive: Positive;

// 编译通过: 通过断言告知Typescript 123是Positive类型
myPositive = 123 as Positive;

// 编译失败：Typescript 无法知道 123 是不是预期的类型
myPositive = 123;

/**
 * Type 'number' is not assignable to type 'Positive'.
  Type 'number' is not assignable to type '{ __brand: "positive"; }'.(2322)
*/

但as的缺点也很明显，断言会导致在开发者不准确的情况下绕过类型错误的提示。比如，即使-1 as Positive可以通过类型检查，但-1明显不是正数。

类型收缩（Type Predicates）

Type Predicates是通过函数的形式，将值类型收缩或断言为某个类型。

type Positive = number & { __brand: "positive" };
 
function isPositive(value: number): value is Positive {
  return value > 0;
}
 
let myPositive: Positive;
 
let value = 123;

if (isPositive(value)) {
  // 编译通过，isPositive函数调用后，Typescript 知道value变量的类型被推断为Positive类型
  myPositive = value;
}
 
// 编译失败，Typescript 无法知道 123 是否是预期的Positive类型
myPositive = 123;
/**
 * Type 'number' is not assignable to type 'Positive'.
  Type 'number' is not assignable to type '{ __brand: "positive"; }'.(2322)
*/

当然，Type Predicates也并不比as来的类型安全。其准确性仍然取决于开发者。

类型断言函数

Type Predicates的另一个缺点是它们需要在代码中写条件判断if （isPositive） { ... }。这些 if 占用更多空间，并增加了比断言as更多的运行逻辑。

为了让代码更易读，我们可以提取一个称为“类型断言函数”的实用函数。类型断言函数接收一个值，如果该值与预期类型不符则抛出错误，否则返回该值。当与Type Predicates配合使用时，类型断言函数可以告知 TypeScript 一个值是Branded types的一个实例类型。

type Positive = number & { __brand: "positive" };
 
function assertPositive(value: number): asserts value is Positive {
  if (value < 0) {
    throw new Error(`${value} is not positive.`);
  }
}

let myPositive: Positive;
 
let value = 123;

assertPositive(value);

// 编译通过，Typescript 知道 value 是Positive类型
myPositive = value;

第三方库

这里罗列一些和Branded types概念相关的，或者提供创建Branded types辅助函数的第三方库。

zod

点击此处查看完整用法:

const Cat = z.object({ name: z.string() }).brand<"Cat">();
const Dog = z.object({ name: z.string() }).brand<"Dog">();
 
type Cat = z.infer<typeof Cat>; // { name: string } & z.$brand<"Cat">
type Dog = z.infer<typeof Dog>; // { name: string } & z.$brand<"Dog">
 
const pluto = Dog.parse({ name: "pluto" });
const simba: Cat = pluto; // ❌ not allowed

io-ts

io-ts提供了运行时类型检查的辅助函数：

import * as t from "io-ts";

// Define a branded codec
interface PositiveBrand {
  readonly Positive: unique symbol;
}

const Positive = t.brand(
  t.number,
  (n): n is t.Branded<number, PositiveBrand> => n > 0,
  "Positive"
);

type Positive = t.TypeOf<typeof Positive>;

// Decode and validate
const result = Positive.decode(42);
// result is Either<Errors, Positive>

ts-brand

ts-brand包提供了社区构建选项，可以共享仅针对Branded types的预写代码：

import { Brand } from "ts-brand";
type Guid = Brand<string, "guid">;
const myKnownGuid = "abc123" as Guid;

effect-ts

Effect是一个流行的用于构建类型丰富的Typescript应用程序框架：

import { Brand } from "effect";
 
type Positive = number & Brand.Brand<"Positive">;
 
const asPositive = Brand.nominal<Positive>();
 
let myPositive: Positive;
 
// Ok: the asPositive tells TypeScript we meant this. 👍
myPositive = asPositive(123);

// Type error: TypeScript doesn't know whether this is intended. 👍
myPositive = 123;
