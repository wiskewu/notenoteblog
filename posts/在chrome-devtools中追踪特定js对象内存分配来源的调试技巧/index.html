在Chrome Devtools中追踪特定js对象内存分配来源的调试技巧

2026/2/26 00:40:489 分钟阅读
devtoolsdebug

在使用Chrome Devtools进行调试时，我们常常会遇到一个对象（比如网络请求的响应对象）的原始调用栈被批处理机制"洗掉"或难以在庞大的代码库中追踪来源的情况。开发者工具上的"内存-时间轴上的分配（Allocations on timeline）"内存分析器虽然有帮助，但存在两个问题：

对象可能因很快被垃圾回收而无法在最终内存视图中捕获； 即使捕获到，在大量匿名object构造函数的实例中也很难定位到具体的目标对象；

解决办法

针对这种情况，可以按照以下步骤进行js对象内存分配来源追踪：

  1. 调试断点：在Sources面板中对应代码块设置断点，让代码在后续操作中当目标对象位于作用域内时暂停；
  2. 准备内存分析：在Memory面板中，开启Allocations on timeline并勾选Allocation stack traces，开始录制分析；
  3. 捕获目标对象：进行页面操作，以让代码进入断点；
  4. 显式"泄漏对象"：在Sources的Scope窗格中，右键目标对象并选择Store as global variable（例如存为temp1）；
  5. 创建唯一标识：在Console控制台中，运行类似temp2 = new class AAAAA { a = temp1 }的代码。这创建一个新的、具有唯一类名（如AAAAA）的全局对象（temp2），并让它引用目标对象temp1；
  6. 从断点处继续执行并停止分析：移除或禁用断点，继续执行脚本，然后停止内存分析；
  7. 定位与溯源：在内存分析结果中，通过类名（Filter by class）搜索唯一标识（如AAAAA），即可精准找到我们创建的那个对象temp2，并查看它（以及它所引用的目标对象temp1）的完整分配堆栈跟踪，从而精确定位其创建源头。

实战示例

进入网站go-view，打开开发者工具的Sources（资源/源码）面板，假设想查看chartEditStore-5ca9db65.js函数createAddHistory参数t数组下的第一个子元素内存分配位置：

createAddHistory(t) {
    this.createStackItem(t, o.ADD, f.CHART)
}

步骤一：调试断点

在对应代码处打上断点。

步骤二：准备内存分析

打开Memory（内存）面板，选择Allocations on timeline（按时间线分配），并勾选Allocation stack traces(more overhead)（记录堆栈信息），然后点击start recording heap profile录制。

步骤三：操作以进入断点

在此例页面中，拖动左侧柱状图至中间画布上，以触发createAddHistory函数执行，进入断点模式。

步骤四：泄漏对象

在断点处的Scope（作用域），选择变量t下的第一个元素（构造函数为经过压缩后的名称hB），右键选择Store as global variable（存储为全局变量）。

此时Console控制台显示新的变量temp1，指向刚才的对象。

步骤五：创建唯一标识

在Console（控制台）中，运行代码temp2 = new class AAAAA { a = temp1 }，此时temp2变量的构造函数为AAAAA，并且属性a执行全局变量temp1。

步骤六：停止录制

在Sources的右侧调试面板中，点击Resume script execution继续执行代码或Deactivate breakpoints停止断点，回到Memory面板，停止内存录制。

获得内存快照。

步骤七：分析内存

在Memory面板，选择刚才的快照Snapshot，在右侧上方的搜索栏Filter by class（通过构造函数过滤）输入框输入AAAAA，执行过滤操作。

根据过滤结果，可以看到属性a指向的构造函数为hB的内存分配位置为index-96a625fd.js。

点击链接index-96a625fd.js，可以看到hB的构造函数代码，此处即为内存分配来源。

总结

在此例中，目标对象的构造函数名为hB，通过这个名称依然可以在内存快照中搜索出对应的内存分配来源信息，但在大型项目中，构造函数名可能并不明确或模糊，则可以使用此方法进行调试定位。
